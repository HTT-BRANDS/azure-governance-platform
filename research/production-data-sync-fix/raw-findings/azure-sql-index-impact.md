# Azure SQL NVARCHAR Widening — Impact on Existing Indexes

**Source**: Microsoft SQL Server T-SQL Documentation (Tier 1)
- https://learn.microsoft.com/en-us/sql/t-sql/statements/alter-table-transact-sql?view=sql-server-ver16#alter-column
- https://learn.microsoft.com/en-us/sql/t-sql/statements/alter-table-transact-sql?view=sql-server-ver16#change-the-size-of-a-column

## Question

The `policy_states` table has indexes on `compliance_state` and `tenant_id`.
Does widening `policy_name` from NVARCHAR(255) to NVARCHAR(1000) affect those indexes?

## Answer: No Impact on Indexes on Other Columns

### ALTER COLUMN Scope

`ALTER TABLE ... ALTER COLUMN` operates on **a single named column**. The operation:
1. Modifies the metadata for that specific column only
2. Does NOT touch any other column's metadata
3. Does NOT rebuild any index that doesn't include the altered column
4. Acquires a table-level Sch-M lock (brief for metadata-only changes)

### What the Docs Say About Indexes

From the "Change the size of a column" section:

> "You can't define the column in an index, unless the column is a **varchar, nvarchar,
> or varbinary** data type and the index isn't the result of a PRIMARY KEY constraint."

This restriction means:
- The **column being altered** can stay in an index IF it's varchar/nvarchar/varbinary
- Since `policy_name` is NVARCHAR, it CAN remain in an index (if it were indexed)
- Indexes on **other columns** (`compliance_state`, `tenant_id`) are irrelevant — they don't reference the altered column

### What Happens to Indexes on the Altered Column

Even if `policy_name` were indexed (it's not in our schema):

| Scenario | Index Impact | Notes |
|----------|-------------|-------|
| Widen NVARCHAR column NOT in any index | **None** | Our case — no index references `policy_name` |
| Widen NVARCHAR column IN a non-clustered index | **None** | NVARCHAR can be widened while indexed |
| Widen NVARCHAR column beyond 900-byte key limit | **Warning only** | Index still works but may have reduced efficiency |
| Change column data TYPE (not just size) | **Index rebuild required** | Not our case |

### Our Specific Table: policy_states

#### Current Indexes

```sql
-- From app/core/database.py
Index("idx_policy_states_compliance", "policy_states", "compliance_state")
Index("idx_policy_states_tenant", "policy_states", "tenant_id")
```

#### Columns Being Altered

| Column | Change | In Any Index? | Impact on Indexes |
|--------|--------|---------------|-------------------|
| `policy_definition_id` | String(500) → String(1000) | **No** | **None** |
| `policy_name` | String(255) → String(500) | **No** | **None** |
| `policy_category` | String(500) → String(1000) | **No** | **None** |

#### Indexes NOT Being Touched

| Index | Column | Impact from ALTER COLUMN on other columns |
|-------|--------|------------------------------------------|
| `idx_policy_states_compliance` | `compliance_state` | **None** — different column |
| `idx_policy_states_tenant` | `tenant_id` | **None** — different column |

### What About the Sch-M Lock?

The Sch-M (Schema Modify) lock is **table-level**, meaning:
- During the lock, ALL queries to the table are blocked (reads AND writes)
- This includes queries using the indexes on `compliance_state` and `tenant_id`
- However, for NVARCHAR widening (metadata-only), the lock is held for **milliseconds**
- Queries in the execution queue will proceed immediately after the lock is released
- No index rebuild, no data rewrite, no log generation beyond the metadata change

### Statistics Behavior

From the documentation:

> **Note**: Statistics that are automatically generated by the query optimizer are
> automatically dropped by `ALTER COLUMN`.

This only applies to **auto-generated statistics on the altered column**:
- If SQL Server auto-created statistics on `policy_name`, those statistics are auto-dropped
- Statistics on `compliance_state` and `tenant_id` are **completely unaffected**
- Auto-dropped statistics are recreated automatically on next relevant query

For **user-created statistics** on the altered column:
- Must be manually dropped with `DROP STATISTICS` before `ALTER COLUMN`
- Our schema doesn't create user statistics on these columns

### Index Key Size Considerations

SQL Server index key maximum is **900 bytes** (450 NVARCHAR characters):

| Column | New Size | In Bytes | Index Key Safe? |
|--------|----------|----------|----------------|
| `policy_name` | NVARCHAR(500) | Up to 1000 bytes | ⚠️ Exceeds 900-byte key limit if indexed |
| `policy_definition_id` | NVARCHAR(1000) | Up to 2000 bytes | ⚠️ Exceeds limit if indexed |

However, since **neither column is currently indexed**, this is purely informational.
If future indexes are needed on these columns, consider:
- Using `INCLUDE` columns instead of key columns
- Creating filtered indexes
- Using hash indexes (if the exact value is always used in WHERE)

## Conclusion

**Widening `policy_name`, `policy_definition_id`, and `policy_category` has ZERO impact
on the existing indexes on `compliance_state` and `tenant_id`.** The operation is:

1. ✅ Metadata-only (no data rewrite)
2. ✅ Sub-millisecond Sch-M lock
3. ✅ No index rebuild on any column
4. ✅ Auto-statistics on altered columns handled automatically
5. ✅ Statistics/indexes on OTHER columns completely unaffected
6. ✅ Safe for production Azure SQL S0
