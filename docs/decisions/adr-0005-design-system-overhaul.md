## TL;DR

The UI feels off-standard because we layered **Tailwind v4 + DaisyUI + Alpine.js** onto a design-token foundation that was already correctly sourced from `~/dev/master-hub-infra`. A thorough audit shows the bolt-ons are **barely used** ‚Äî so the overhaul is closer to **3‚Äì4 days of focused work** than the 2‚Äì3 weeks originally estimated.

**Action:** swap the build engine to Tailwind v3 standalone CLI, delete unused deps, unify CSS into a single `design-tokens.css` + `design-utilities.css` layer, adopt the `Domain-Intelligence` dark-glassmorphism surface vibe, and add Jinja macros for the 5 most-used ds-template primitives.

## 1. Audit Findings (2026-04-17)

### 1.1 We're already 80% on-standard

`app/core/css_generator.py` header literally says:
> *"Sourced from ~/dev/microsoft-group-management design system."*

All 189 custom properties in `theme.src.css` trace back to the master-hub-infra token vocabulary. The brand hex `#500711` + all 10-stop Fluent ramps + Walmart/Riverside `--color-wm-*` scale + per-tenant `[data-brand]` override pattern are **correct and keep-worthy**. This is not a rewrite, it's a re-plumbing.

### 1.2 What's actually broken

| Problem | Evidence | Fix difficulty |
|---|---|---|
| Off-standard build stack | Tailwind v4 + DaisyUI + `@tailwindcss/cli` (no other HTT app uses this) | **Easy** ‚Äî swap to Tailwind v3 standalone binary |
| Dead code bloat | `theme.css` is 185KB / 6,358 lines of **generated output** from v4 ‚Äî mostly unused DaisyUI + utility classes | **Easy** ‚Äî auto-regenerated by new build |
| Alpine.js dead weight | **0 template uses** of `x-data`/`x-show`/`x-on` | **Trivial** ‚Äî delete the `<script>` tag |
| DaisyUI minimal adoption | Only `menu` class (3 occurrences, 7 of 42 templates) | **Trivial** ‚Äî replace with Tailwind utilities |
| No visual surface system | Current UI is flat light-mode ‚Äî Domain-Intelligence has a mature dark-glassmorphism vibe that's distinctly on-brand | **Medium** ‚Äî port the SWA surface tokens |
| Jinja components are feature-specific | `cost_summary_card.html`, `sync_status.html` ‚Äî no `Card`, `StatCard`, `PageShell` primitives | **Medium** ‚Äî add 5 macros |

### 1.3 What's surprisingly NOT broken

- Brand tokens are correct (`#500711` burgundy, Inter font, colorblind-safe semantic colors)
- `css_generator.py` + `design_tokens.py` modules produce per-tenant CSS cleanly
- Templates use plain Tailwind utilities that work identically in v3 (`p-4`, `bg-brand-primary-50`, `text-success`)
- WCAG-friendly text colors (`--text-muted: #57606A` = 5.74:1 on white)
- Dark mode class toggle (`.dark`) already wired

## 2. Target Architecture

### 2.1 CSS layering (mirrors Domain-Intelligence)

```
app/static/css/
‚îú‚îÄ‚îÄ design-tokens.css      ~200 lines  ‚Üê unified :root tokens (replaces theme.src.css :root block)
‚îú‚îÄ‚îÄ design-utilities.css   ~100 lines  ‚Üê focus rings, .dark overrides, brand utility helpers
‚îú‚îÄ‚îÄ tailwind-output.css    ~30-50KB    ‚Üê built by Tailwind v3 standalone CLI
‚îî‚îÄ‚îÄ (delete: theme.css, theme.src.css, ui-polish.css, riverside.css, accessibility.css)
```

Net: **237KB ‚Üí ~50KB** of CSS shipped.

### 2.2 Build pipeline

**Before:**
```
npm install ‚Üí npx @tailwindcss/cli v4 ‚Üí theme.src.css ‚Üí theme.css (185KB)
```

**After:**
```
scripts/build-css.sh ‚Üí tailwindcss v3 standalone binary ‚Üí tailwind-output.css
```

- **No Node runtime required.** Tailwind v3 ships single-binary releases for macOS/Linux/Windows
- `package.json` reduces to... nothing (can delete it entirely; no JS deps remain)

### 2.3 Template primitives (Jinja macros)

Add `app/templates/macros/ds.html` with 5 primitives ported from `~/dev/master-hub-infra/src/components/ds-template/`:

| Macro | React source | Purpose |
|---|---|---|
| `ds.page_shell(title, subtitle, breadcrumb)` | `PageShell.tsx` | Standard page chrome |
| `ds.card(title, body, footer)` | `Card.tsx` | Content container |
| `ds.stat_card(label, value, trend, icon)` | `StatCard.tsx` | Dashboard KPI tiles |
| `ds.alert(message, severity)` | `Alert.tsx` | Info/warning/error banners |
| `ds.button(label, variant, icon)` | `Button.tsx` | Primary/secondary/ghost buttons |

*Not porting right now:* DataTable, Modal, Tabs, FormField, Toolbar ‚Äî add on-demand as pages need them.

### 2.4 Visual surface system

Adopt Domain-Intelligence's SWA Dark Glassmorphism tokens (already keeps our light mode intact):

```css
--bg-gradient:    linear-gradient(135deg, #0a0a0f 0%, #0d1117 50%, #111827 100%);
--surface:        #161b22;
--surface-glass:  rgba(22, 27, 34, 0.7);
--swa-radius:     12px;     /* card corners */
--swa-shadow:     0 4px 24px rgba(0, 0, 0, 0.3);
--swa-transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1);
```

This is the specific vibe that distinguishes HTT dark mode from a generic dark theme.

## 4. Risks + Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| `css_generator.py` breaks under new CSS structure | Low | It outputs `[data-brand]` overrides as strings ‚Äî doesn't care about the build tool |
| Some template uses a v4-only class | Low | Audit found utilities only; worst case = fix 2‚Äì3 instances |
| Chart.js colors break (reads CSS vars) | Low | We preserve the dual `--brand-primary` + `--brand-primary-rgb` pattern |
| "DaisyUI `menu` class" in 3 spots | Low | Replace with `flex flex-col space-y-1` + matching utilities |
| Tailwind v3 standalone binary unavailable in CI | Low | It's a GitHub-hosted static binary, already downloadable (verified HTTP 200) |
| Visual regression on pages we don't touch in Phase 3 | Medium | Accepted ‚Äî they still use valid Tailwind utilities and the new tokens; will look slightly different but not broken |
